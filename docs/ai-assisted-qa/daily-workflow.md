# AI in Daily QA Workflow

## How AI Has Changed My Day-to-Day

I didn't adopt AI tools all at once. I started using them for specific problems — understanding code I didn't write, drafting test cases, explaining error messages — and over time I've extended that into more parts of my work.

Here's an honest account of where AI fits in my current workflow:

---

## Documentation and Writing

The most consistent daily use is documentation. Writing test plans, bug reports, release notes, and process documentation is necessary but time-consuming. I've found that AI drafts are faster to edit than writing from scratch, especially for documentation that follows a predictable structure.

I write the thinking; AI helps me express it clearly and quickly. I review everything before it goes anywhere.

---

## Requirement Analysis

When a new user story or specification lands, I read it with the question: what's missing? Where is this ambiguous? What will be hard to test?

I've started using AI as a second pass on this — paste the spec, ask "what edge cases or unclear requirements do you see?" The output isn't always useful, but it sometimes surfaces something I hadn't noticed, and the exercise of evaluating each suggestion makes my own analysis more systematic.

---

## Code Explanation and Review

I'm not a strong programmer, and I work alongside engineers who write Python, Kotlin, and Swift. When test code or product code is doing something I don't fully understand, AI explanation is faster than pulling someone away from their work.

I've also used AI to review my own test code — asking whether a particular fixture design is idiomatic, whether a test I've written will actually catch what I think it will catch.

---

## Learning Unfamiliar Territory

When I need to understand something technical quickly — how a particular API authentication scheme works, what a Kotlin coroutine does, what the difference is between two similar Appium strategies — AI is a faster starting point than documentation, especially for things where I need the conceptual understanding more than the implementation details.

---

## What AI Hasn't Changed

The judgment calls in QA work still come from experience and context that AI doesn't have. Deciding what risk is acceptable for a release, knowing when a bug is a blocker versus a known issue, understanding the history of a particular subsystem — none of that is something I can outsource.

The best framing I've found: AI handles the parts of QA work that are about processing and expression. The parts that are about judgment, context, and domain knowledge remain mine.
