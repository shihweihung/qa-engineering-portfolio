# 與 DQA 的合作

## 角色定位

QA 和 DQA 是不同的職能，QA 不管理 DQA。

DQA 負責設計品質驗證，包含 EVT、DVT、PVT 各階段的規劃、可靠性測試、Validation Plan 制定，以及硬體層面的 Root Cause Analysis。System QA 則負責整機功能與使用者體驗的驗證：在不同 Build 與使用情境下，產品的行為是否正確。

兩個角色是互補的。當 System QA 發現的問題可能涉及硬體或可靠性時，重點不是釐清責任歸屬，而是把 DQA 需要的資訊完整提供出去，讓分析可以快速展開。

---

## QA 需要提供什麼

QA 不需要看懂示波器波形或 PMIC 電路圖。但在把問題移交給 DQA 之前，資訊的品質很重要。

在升報之前，我會先確認幾件事：

- 問題可以穩定重現嗎？
- 從哪個 Build 開始出現？
- 是特定機器，還是所有機器都會發生？
- 是特定硬體版本，還是跨版本都有？
- 是特定使用情境，還是一般操作就會觸發？

確認之後，提供完整的重現條件：Build Version、Hardware Revision、Serial Number、零件或 Lot 資訊（如適用）、操作步驟、Log、以及行為可見的話附上影片。

DQA 能在自己的環境快速重現問題，分析就能快速推進。資訊不完整是跨職能問題移交最常見的瓶頸。

---

## 不懂硬體，怎麼看 DQA 報告

QA 不需要看懂 DQA 產出的每一份報告。重要的是知道哪類問題對應哪類報告，以及怎麼看結論。

大多數情況下，我只需要確認四件事：

1. **Test Condition** — 驗證時使用的環境與情境
2. **Specification** — 產品需要符合的規格
3. **Test Result** — 實際量測的結果
4. **Conclusion** — 對照規格的 PASS 或 FAIL

如果結論是 FAIL，後續確認的方向是：Root Cause 是什麼、影響範圍有多大、Corrective Action 是什麼、Verification Plan 怎麼規劃。

報告裡其他的內容是給硬體團隊判斷用的。在沒有足夠領域知識的情況下試圖解讀，產生的噪音通常多過有用的洞察。

## 「DVT 為什麼沒有發現這個問題？」

這個問題在量產後出現問題時很常被提出。直覺是認為前期驗證不夠完整，但這通常不是對的起點。

在得出「DVT 漏掉了」這個結論之前，應該先確認：

- 當時的 Validation Plan 是否包含這個情境？
- 驗證當下產品是否符合 Spec？
- DVT 之後 Firmware 是否更新過？
- DVT 之後是否更換了零件或供應商？
- 是否新增了當初驗證計畫沒有涵蓋的使用情境？

常見的模式是：DVT 階段的待機電流測試 PASS，後續 Firmware 更新加入了 Background Sync 功能，上市後使用者發現待機時間明顯縮短。DVT 的結果沒有問題，是產品在驗證之後改變了，但對應的 Power Regression 沒有重新執行。

更有用的問題不是「DVT 為什麼沒有抓到」，而是「中間發生了什麼變更，哪些驗證應該要被觸發但沒有被觸發？」

---

## 把問題框架成風險，而不是指責

QA 怎麼描述一個問題，決定了接下來的對話怎麼走。

說「這個設計有問題」會讓對方進入防禦狀態。換成「目前只有室溫的測試結果，低溫使用情境還沒有完成驗證，而產品規格有包含低溫使用 — 建議在 Release 前確認是否需要補驗」，同樣的疑慮就變成一個需要被處理的風險。

第二種說法比較難被忽略，也比較容易推動行動，而且不需要任何人在事實釐清之前先承認錯誤。用證據和風險語言說話，事情比較會往前走。用指責語言說話，產生的是阻力。

這不是在軟化訊息，而是讓訊息變得有用。
