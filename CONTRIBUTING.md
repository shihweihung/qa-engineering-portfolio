# Contributing Guide

Thank you for contributing to the QA Interview Bible documentation site.

## Documentation Principles

- Use Markdown only.
- Keep one topic per page.
- Keep filenames lowercase.
- Use descriptive headings.

## Required Article Structure

Every article should include the following sections:

1. Summary
2. Main Content
3. Interview Questions
4. Real Example
5. References

## Writing Rules

- Start each page with a clear title that matches the topic.
- Keep sections focused and practical.
- Prefer examples that can help with interview preparation, QA work, or technical learning.
- Add references for external sources, related pages, or official documentation when relevant.

## File Naming

- Use lowercase letters only.
- Separate words with hyphens when needed.
- Avoid spaces and special characters.

Examples:

- `api-testing-basics.md`
- `mobile-release-checklist.md`
- `bug-reproduction-guide.md`

## Page Template Expectation

Use this structure when creating a new article:

```md
# Page Title

## Summary

Brief overview of the topic and why it matters.

## Main Content

Core explanation, steps, framework, or notes.

## Interview Questions

- Question one
- Question two

## Real Example

Describe a realistic scenario, bug, project, or experience.

## References

- Related internal page
- Official documentation
```

## Quality Checklist

Before submitting changes, confirm the page:

- Uses Markdown only
- Covers exactly one main topic
- Uses a lowercase filename
- Includes descriptive headings
- Includes all required article sections
